# LanTalk

LanTalk 是一个基于 C# / .NET 10、Avalonia UI 和 SukiUI 的局域网即时通信客户端。它面向教室、实验室、小型办公室等局域网协作场景，目标是“无需登录注册、打开即用、自动发现、能私聊、能广播、能传文件、能保存记录”，并逐步演进为完整、强大的内网协作应用。

## 技术栈
- C# / .NET 10
- Avalonia UI 12
- SukiUI 7
- CommunityToolkit.Mvvm
- UDP + TCP Socket
- SQLite
- System.Text.Json Source Generator

## 功能状态

### 已完成
- 局域网自动发现、在线用户列表、一对一私聊、全员广播。
- 临时群组、永久群组、群组文本、群组图片/文件、离线补发。
- 单文件、多文件、文件夹传输与断点续传。
- 私聊文本端到端加密、群组文本逐成员端到端加密。
- 附件受保护传输：已协商 E2EE 的会话可使用加密元数据与 AES-GCM 流式附件传输；旧端或未协商会话保持兼容路径。
- 聊天记录搜索、聊天记录导出（CSV / JSON）、文件传输记录导出（CSV）。
- 联系人分组、最近会话排序、多会话切换、未读提醒、消息撤回、已读回执。
- 系统托盘、桌面通知回退提醒、Windows 开机自启设置、主题切换和生产日志。

## 已实现功能
- Telegram 风格双栏主界面。
- 本机昵称、部门/分组、端口、自动发现网段、文件接收目录、明暗主题的 JSON 设置保存。
- 联系人按部门/分组展示，部门信息会随 UDP 自动发现同步，并持久化到 KnownUsers。
- 最近会话与联系人搜索过滤。
- 最近会话按未读优先、最近活跃时间排序。
- 多会话切换：支持上一会话、下一会话、下一未读会话按钮和快捷键。
- 多人会话：支持创建临时群组和永久群组，群组消息点对点发送给在线成员。
- 群组文本离线补发：发送给离线成员的群组文本会进入本地待投递队列，对方重新上线后自动补发。
- 图片消息：支持单独选择图片发送、聊天内缩略图预览和大图缩放查看。
- 表情输入：输入栏提供常用表情快捷插入。
- 头像体验：联系人和本机头像按昵称生成首字与稳定色彩。
- 当前会话内聊天记录搜索。
- 系统托盘：关闭窗口后保持后台运行，可从托盘恢复或退出。
- 桌面通知：窗口隐藏、最小化或非活跃时收到消息/文件请求会弹出提醒，并支持“优先尝试系统通知，失败后回退应用内 toast”的配置。
- 未读提醒：会话未读数、窗口标题和托盘提示同步显示。
- 已读回执：私聊显示“未读/已读”，群组消息按成员统计“已读 x/y”。
- 消息撤回：发送方可撤回自己发送的私聊/群组文本和图片，接收端同步显示撤回提示。
- 启动时端口占用检测和左侧本机在线状态显示。
- UDP 局域网自动发现：HELLO、ONLINE、HEARTBEAT、BYE，支持自定义发现网段。
- 在线用户列表实时更新。
- TCP 私聊消息发送/接收。
- 私聊文本端到端加密：会话内可手动开启，使用临时 ECDH 密钥协商和 AES-GCM 加密消息内容。
- 群组文本消息：通过 TCP 逐个发送给群组在线成员，永久群组会保存并在重启后恢复。
- 群组文本端到端加密：群组会话可开启逐成员加密，复用一对一 E2EE 会话分别加密 `GroupMessage`，未协商或离线成员进入加密补发队列。
- 群组图片/文件：复用现有 TCP 文件传输通道，按在线成员逐个发送请求，并在群组会话中聚合显示投递状态。
- 附件保护：已协商 E2EE 的接收方支持受保护附件时，文件请求会携带加密元数据，TCP 文件流使用 AES-GCM 分块加密传输。
- 全员广播消息发送/接收。
- SQLite 聊天记录与文件传输记录。
- 单文件、多文件和文件夹传输：支持接收/拒绝确认、文件夹相对目录结构保留、流式分块传输、进度显示。
- 断点续传：传输中断后重新发送同一文件或同一批次时，接收端会返回已有字节数，发送端从断点继续发送。
- 离线文件提醒：对方离线时，文件、图片、多文件和文件夹请求会进入本地队列；对方重新上线后先收到离线文件提醒，再走原有接收确认和流式传输。
- 附件压缩：设置中可开启发送前 GZip 压缩；已启用附件加密时自动跳过压缩以保持兼容。
- 导出工具：当前会话可导出为 CSV / JSON，文件传输记录可导出为 CSV。
- Windows 开机自启：设置中可开关，使用当前用户 `Run` 注册表项。
- 生产日志：控制台与 `%AppData%\\LanTalk\\logs` 双写，按天滚动并带文件大小保护。
- 文件接收目录不可用时显示失败状态并保存失败记录。
- 面向 Native AOT 的 JSON Source Generator 和轻量依赖结构。

## 运行方式
```bash
dotnet build LanTalk.sln
dotnet run --project src/LanTalk.App
```

## 测试方式
```bash
dotnet test LanTalk.sln
```

## 发布方式
```bash
dotnet publish src/LanTalk.App/LanTalk.App.csproj -c Release -r win-x64 --self-contained false -v:minimal
```

Native AOT 发布：
```bash
dotnet publish src/LanTalk.App/LanTalk.App.csproj -c Release -r win-x64 -p:PublishAot=true -v:minimal
```

发布目录：
- `src/LanTalk.App/bin/Release/net10.0/win-x64/publish/`

详细发布说明见 [docs/release.md](docs/release.md)。

当前 Native AOT 可生成 `win-x64` 发布产物；Avalonia DataGrid 依赖仍会产生 trim / AOT 分析警告，后续 UI 深度打磨时需要继续观察。

## 演示流程
1. 在同一局域网的两台电脑启动 LanTalk。
2. 等待左侧在线用户列表自动出现对方。
3. A 选择 B，发送私聊文本。
4. B 回复 A。
5. A 进入“全员广播”，发送广播消息。
6. B 向 A 发送文件请求，A 选择接收或拒绝。
7. 接收文件时观察进度，完成后到设置中的接收目录查看文件。
8. 重启应用，打开会话确认历史记录仍可加载。
9. 在右侧聊天记录搜索框输入关键词，确认能检索当前会话历史消息。
10. 修改部门/分组后重启两端，确认联系人会按部门分组展示。
11. 选择图片发送，接收方确认后在聊天气泡中预览图片，并点击缩略图进行放大/缩小查看。
12. 点击“新建群组”，选择成员并创建临时群组或永久群组，在群组会话中发送多人消息。
13. 在群组会话中发送图片或文件，确认在线成员收到来自群组的接收请求，发送方看到群发进度汇总。
14. 在群组会话中开启端到端加密，确认在线成员会自动发起一对一加密协商；未完成协商的成员不会收到明文群文本，而是进入加密补发队列。
15. 点击文件按钮一次选择多个文件，接收方只确认一次，确认后批量传输。
16. 点击文件夹按钮选择一个目录，接收方确认后在接收目录中看到保留相对路径的文件夹内容。
17. 传输大文件时中断连接，再次发送同一文件或批次，确认进度从已接收字节附近继续。
18. A 给 B 发送私聊消息，B 打开会话后，A 端消息状态从“未读”变为“已读”。
19. A 在群组中发送消息，B/C 打开群组后，A 端显示“已读 1/2”“已读 2/2”。
20. A 撤回一条私聊或群组消息，B/C 端显示“对方撤回了一条消息”。
21. A 向离线 B 发送文件或文件夹，B 重新上线后先收到离线文件提醒，再收到可确认的文件请求。
22. A 与 B 在私聊中开启端到端加密后发送图片或文件，确认接收方可正常接收，且旧端 / 未协商路径不被破坏。
23. 点击“导出聊天”，将当前会话导出为 JSON 或 CSV；点击“导出传输”，导出文件传输记录 CSV。
24. 在设置中开启“发送附件前尝试 GZip 压缩”和“Windows 开机自启”，保存后确认设置持久化。

## 常见问题
- 如果端口被占用，请在设置面板中修改 UDP/TCP 端口后重启 LanTalk，或关闭占用端口的程序。
- 自动发现网段默认使用 `Auto`；也可以在设置中逐条添加 `192.168.1.0/24`、`192.168.1.*` 或 `192.168.1.255` 后重启。
- Windows 防火墙可能拦截 UDP 广播或 TCP 连接，演示前需要允许应用通过专用网络。
- 点击窗口关闭按钮会将 LanTalk 最小化到系统托盘；需要完全退出时请使用托盘菜单中的“退出”。
- 多实例在同一台电脑上默认不能同时使用 50000/50001/50002，需要修改端口后再测试。
- 群组当前支持文本、图片、文件、多文件和文件夹，支持群组文本与附件离线补发；群组文本加密已稳定，附件保护会在对端支持且一对一 E2EE 已协商时启用。
- 如果系统通知不可用，LanTalk 会自动回退到应用内桌面 toast；这不会影响消息接收。
- 并行执行多个 `dotnet build` / `dotnet test` / `dotnet publish` 时，Windows 上可能出现文件锁；建议始终串行执行验证命令。
